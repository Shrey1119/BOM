# Team Member 3 Runbook: Compliance & Enrichment Specialist

This runbook describes the automated workflow for enriching a raw SBOM JSON file (produced from source scanning) with all **21 client-required attributes** and performing the final compliance validation check.

---

## Roles & Responsibilities
As the **Compliance & Enrichment Specialist**, you are responsible for:
1. Configuring metadata defaults and custom exceptions in [config.json](file:///d:/College%20Work/Internship/BOM/Member_3/config.json).
2. Running the enrichment tool [enricher.py](file:///d:/College%20Work/Internship/BOM/Member_3/enricher.py) to map vulnerability scans, query package registries (like PyPI), and populate missing lifecycle attributes.
3. Formatting Package URLs (PURLs) for all scanned dependencies to ensure proper unique identification.
4. Running the validation script [validator.py](file:///d:/College%20Work/Internship/BOM/Member_3/validator.py) to guarantee that all 21 mandatory attributes are present before delivery.

---

## 21-Attribute Mapping Details
Our automation tool maps the required attributes into the **CycloneDX v1.6** JSON schema as follows:

| Attribute # | Attribute Name | CycloneDX Mapping Location | Source / Resolution Method |
|---|---|---|---|
| **1** | Component Name | `components[].name` | Extracted by scanner |
| **2** | Component Version | `components[].version` | Extracted by scanner |
| **3** | Component Description | `components[].description` | PyPI JSON API / Fallback to metadata |
| **4** | Component Supplier | `components[].supplier.name` | PyPI API author field / Config overrides |
| **5** | Component License | `components[].licenses[].license.name` | PyPI API license field / Config overrides |
| **6** | Component Origin | Property: `origin` | Set as `open-source` (PyPI) or overridden |
| **7** | Component Dependencies | `dependencies` root array | Extracted from package lockfiles |
| **8** | Vulnerabilities | `vulnerabilities` root array | Analyzed by Trivy CLI scanner |
| **9** | Patch Status | Property: `patch_status` | Compares current version with PyPI's latest release |
| **10** | Release Date | Property: `release_date` | Upload date fetched from PyPI API history |
| **11** | End-of-Life (EOL) Date | Property: `eol_date` | Calculated as `Release Date + offset` / Overrides |
| **12** | Criticality | Property: `criticality` | Resolved via `config.json` overrides (default: `medium`) |
| **13** | Usage Restrictions | Property: `usage_restrictions` | Resolved via `config.json` overrides (default: `None`) |
| **14** | Checksums or Hashes | `components[].hashes` | SHA-256 hashes generated by scanner |
| **15** | Comments or Notes | Property: `comments` | Resolved via `config.json` overrides (default: `Scanned`) |
| **16** | Author of SBOM Data | `metadata.authors` | Hardcoded inside `config.json` |
| **17** | Timestamp | `metadata.timestamp` | Generated dynamically at assembly time (ISO 8601) |
| **18** | Executable Property | Property: `executable` | Set to `false` for package dependencies |
| **19** | Archive Property | Property: `archive` | Set to `true` for PyPI tarballs/wheel packages |
| **20** | Structured Property | Property: `structured` | Format descriptor: `CycloneDX JSON v1.6` |
| **21** | Unique Identifier | `components[].purl` | strictly formatted: `pkg:pypi/{name}@{version}` |

## Detailed Step-by-Step Execution Guide

### Step 1: Pre-requisites and Workspace Verification
Ensure that python is available in your execution environment. You do not need any third-party pip dependencies, as `enricher.py` and `validator.py` utilize Python's built-in standard libraries (`urllib` and `json`).

Verify that your directory contains:
- `config.json` (defines mapping policies)
- `enricher.py` (performs the API data injection)
- `validator.py` (validates the final schema)

### Step 2: Configure Attributes Override Map
Open [config.json](file:///d:/College%20Work/Internship/BOM/Member_3/config.json) to declare custom criticality ratings, licensing notes, or suppliers for components.
Example override structure:
```json
"overrides": {
  "requests": {
    "supplier": "Kenneth Reitz & Requests Community",
    "criticality": "high",
    "comments": "Core network HTTP communications module."
  }
}
```

### Step 3: Run the Enrichment Script
After Team Member 2 runs the scanner and outputs the raw SBOM file (e.g. `sbom_raw.json`), run the following command to query package registries and enrich the fields:

```powershell
# Command syntax: python enricher.py [path_to_raw_sbom] [path_to_enriched_output]
python Member_3\enricher.py sbom_raw.json sbom_enriched.json
```

This will:
- Parse all dependencies in `sbom_raw.json`.
- Fetch release dates and license parameters from PyPI online registry.
- Map the standard/custom attributes.
- Save the completed document to `sbom_enriched.json`.

### Step 4: Run the Compliance Validation Check
Run the validator script to guarantee zero compliance errors. The validator checks the presence and format of all 21 parameters for all dependencies.

```powershell
# Command syntax: python validator.py [path_to_enriched_sbom]
python Member_3\validator.py sbom_enriched.json
```

If validation is successful, the terminal will print `SUCCESS: Enriched SBOM is fully compliant with all 21 client attributes`. If it fails (returns exit code `1`), check the report table printed on screen to resolve any missing mappings in `config.json`.
