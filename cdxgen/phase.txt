Phase 1: START (Tooling Architecture & Setup)
In this foundational phase, you prepare your environment to handle cdxgen's advanced code parsing capabilities
.
Install cdxgen and IR Dependencies: Install the cdxgen utility. Unlike basic scanners, cdxgen relies on an intermediate representation (IR) toolkit called atom and the chen (Code Hierarchy Exploration Net) library to structurally understand your code
. Ensure these are available in your environment.
Prepare Post-Processing (sbomify-action): While cdxgen is incredibly advanced at tracing code, it still does not natively know a supplier's business data. You must retain sbomify-action in your pipeline to fulfill the client's lifecycle requirements (Release Dates, EOL Dates, Supplier, and License details)
.
OpenAPI Specifications: If your source code is a web application, gather your OpenAPI specification files. cdxgen can consume these to understand how specific HTTP endpoints connect to your code
.
Phase 2: PROGRESS (Deep Source Code Scanning & Enrichment)
This is the execution phase where cdxgen evaluates your local source code folder to build the baseline SBOM
.
Step 1: Execute in Deep Mode. Run cdxgen against your source code folder using the --with-reachables flag. This triggers "deep mode," allowing the tool to perform static slicing on your application
.
Step 2: Trace the Execution Paths. cdxgen will use Framework-Forward Reachability (FFR) and Semantic Reachability to trace exact variable assignments and function calls from your application's entry points down to external library sinks
. This proves whether your code actually touches a vulnerable dependency.
Step 3: Dynamic Profiling (Optional). If you are able to run the application locally, you can use cdxgen's tracebom command. This dynamically profiles the running application to capture dynamically resolved software components and negotiated TLS cipher suites
.
Step 4: Attribute Enrichment. Pass the highly detailed CycloneDX output document through sbomify-action to inject the missing business data and map the Unique Identifiers (Attribute 21) using the Package URL (PURL) scheme required by CERT-In
.
Phase 3: ADVANCE (Exploitability Management & Delivery)
This final phase leverages the unique intelligence gathered by cdxgen to eliminate wasted patching efforts and securely deliver the artifacts to the client
.
Step 1: Evaluate Callstack Evidence. Look at the generated CycloneDX document. cdxgen serializes the callstack evidence directly into the artifact
. If a vulnerability exists in a downloaded library, but cdxgen proves that your application's code path never actually reaches or invokes that vulnerable function, your security team can definitively deprioritize the alert
.
Step 2: Generate Automated VEX Documents. Using platforms like OWASP Dependency-Track, you can feed in this reachability evidence to automatically generate a Vulnerability Exploitability eXchange (VEX) document
.
Step 3: Flag as "Not Affected". For all unreachable vulnerabilities, you will flag them in the VEX document as "Not Affected" and provide the client with the precise technical justifications generated by cdxgen
. This satisfies the vulnerability requirements without forcing your developers to patch code that poses no actual threat.
Step 4: Attestation. Cryptographically sign the final SBOM and VEX documents using tools like Cosign or the in-toto framework to prove the integrity of the data before delivering it to the client's admission controllers
