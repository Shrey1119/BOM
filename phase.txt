Team Member 1 (Security & Governance Lead): Focuses on secure storage, vulnerability management (VEX), and final secure delivery.
Team Member 2 (Tooling & Execution Engineer): Focuses on running the Trivy scanner, targeting the source code, and generating the baseline data.
Team Member 3 (Compliance & Enrichment Specialist): Focuses on sbomify-action, data enrichment, and validating the 21 specific client attributes.
Phase 1: START (Foundational Activities)
This phase lays the groundwork for your scanning process and sets up the required tools

Team Member 1 (Lead): Identify Security Requirements & Storage. You are responsible for determining the SBOM format (must be CycloneDX or SPDX) and setting up secure, access-controlled storage for the generated SBOMs before they are sent to the client

Team Member 2 (Engineer): Target the Code & Install Trivy. Identify the specific source code folder that needs to be scanned
 Install and configure the Trivy CLI on the scanning environment, ensuring its continuous vulnerability scanning engine is active

Team Member 3 (Specialist): Tooling for Missing Attributes. Since Trivy cannot natively find business lifecycle data, you must install and configure the open-source sbomify-action toolkit
 Review the client's 21 attribute requirements to understand exactly which fields this tool will need to query from external registries

Phase 2: PROGRESS (Building upon the Foundation)
This phase involves the actual execution of the scan and the enrichment of the data

Team Member 2 (Engineer): Baseline Generation & Vulnerability Scan. Run the Trivy scan directly against the target source code folder. This step will automatically generate the structural metadata (Component Name, Version, Origin, Dependencies, Checksums)
 Crucially, Trivy will also fulfill Attribute 8 (Vulnerabilities) by attaching severity ratings and CVE identifiers to the components
 Pass the output file to Team Member 3.
Team Member 3 (Specialist): Post-Processing & Attribute Enrichment. Pass Team Member 2's Trivy output through the sbomify-action tool
 This will automatically inject the missing lifecycle data required by the client: Release Date (Attribute 10), End-of-Life Date (Attribute 11), Supplier (Attribute 4), and License (Attribute 5)
 You must also format Attribute 21 (Unique Identifier) using the exact Package URL (PURL) scheme: pkg:supplier/OrganizationName/ComponentName@Version

Team Member 1 (Lead): Internal Mapping. Create an internal mapping of the generated SBOM for your organization's own records to track what components are actively being used in this build

Phase 3: ADVANCE (Mature & Scalable Delivery)
This final phase involves validating the output, handling discovered vulnerabilities, and securely sharing the artifact with the client

Team Member 3 (Specialist): Final 21-Attribute Validation. Conduct a final manual or automated review of the enriched SBOM. Ensure all remaining structural and authorship fields are populated, including Criticality (Attribute 12), Executable/Archive Properties (Attributes 18-19), and the Author of SBOM Data & Timestamp (Attributes 16-17)

Team Member 1 (Lead): VEX & CSAF Generation. If Trivy flagged any vulnerabilities, you must evaluate them and create a Vulnerability Exploitability eXchange (VEX) document for the client
 You will classify each vulnerability into one of four statuses: Not affected, Affected, Fixed, or Under Investigation
 Follow this up with a Common Security Advisory Framework (CSAF) advisory detailing any mitigation steps

Team Member 2 (Engineer): Secure Distribution. You cannot share raw SBOMs openly. Split the final document into a Public SBOM (non-sensitive info) and a Private SBOM (confidential/vulnerability info)
 Cryptographically sign the documents to ensure integrity, and use secure file-sharing platforms (like HTTPS or RBAC-enabled portals) to deliver the final artifacts to the client
